# InternalWebServer

## Overview

InternalWebServer is the front door of MitchellNET, providing NGINX reverse proxy, SSL termination, and a static web UI for all services running on the Ubuntu server at 192.168.2.10. It routes all traffic under `https://mitchellnet.local` to the appropriate backend containers via the `mitchellnet` Docker network, acting as the single ingress point for the home lab.

## NGINX Routing Table

| Path Prefix | Routes To | Notes |
|-------------|-----------|-------|
| `/` | static web root | HTML/CSS/JS served directly by nginx-proxy |
| `/fitness/` | `fitness-tracker:5000` | `proxy_pass http://fitness-tracker:5000/;` — trailing slash intentional (Approach A; see mitchellnet-infra runbook). Served from standalone container. |
| `/api/bench/` | `bench-instrument-service:8000` | `proxy_read_timeout 120s` |

## SSL

SSL terminates at NGINX using mkcert certificates for `mitchellnet.local`. Certificates are generated by `mitchellnet-infra/ssl/generate.sh` and mounted into the NGINX container at `/etc/nginx/ssl/`. All backend services communicate over plain HTTP within the `mitchellnet` Docker network — encryption is only needed at the edge.

---

## Docker Networking

`nginx-proxy` must be connected to the `mitchellnet` external Docker network for upstream hostname resolution to work. This is configured permanently in [docker-compose.yml](docker-compose.yml):

```yaml
networks:
  mitchellnet:
    external: true
```

The `mitchellnet` network must exist on the host before deploying. Create it once with:

```bash
docker network create mitchellnet
```

If the network does not exist, `docker compose up` will fail and `nginx-proxy` will be unable to reach any upstream service.

## Deployment

- Runs on the Ubuntu server at 192.168.2.10 (2019 iMac, `iMac19,1`).
- Deployed automatically via GitHub Actions on merged PR using the self-hosted runner `imac-server-runner`.
- The deploy workflow syncs files via rsync, restarts containers, and verifies the deployment.
- SSL certificates are excluded from rsync and must be present on the server before the first deploy.

Monitor deployments in the [Actions tab](https://github.com/theAgingApprentice/InternalWebServer/actions).

## Adding a New Service

Two steps are required to add a new service to MitchellNET:

1. Add a `location` block to [nginx/conf.d/prod.conf](nginx/conf.d/prod.conf) pointing to the new container name and port.
2. Commit and deploy via pull request.

For the full routing design, see `mitchellnet-infra/docs/ARCHITECTURE.md`.
