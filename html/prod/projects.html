Project Best Practices & Guidelines

This page contains a set of best practices and guidelines to help ensure that Aging Apprentice projects are well-organized, maintainable, and successful.

GitHub Repositories vs. Projects

The *Repository* is the core unit on GitHub. It stores project's code, files, commit history, issues, pull requests, and more. Each repo is typically dedicated to a single codebase or component. The *Project* is a flexible planning and tracking tool (like a Kanban board, table, or roadmap) for organizing work. Projects live at the user, organization, or repository level and manage issues and pull requests (PRs). Projects do not contain code or repositories—they reference issues and PRs from repositories.

How Repositories Are Linked to Projects

  • In GitHub Projects, repositories are not directly "linked" like in classic projects.
  • Linkage occurs indirectly by adding individual issues and pull requests from any accessible repository to the project.
  • When creating a new item in a project, you can select the target repository to create the issue/PR there.
  • Draft issues can exist in the project without belonging to any repository initially.

Many-to-Many Relationships

  • One project → many repositories: Yes. A single project can include issues and PRs from multiple repositories, ideal for cross-repo coordination (e.g., microservices or multi-component apps).
  • One repository → many projects: Yes. Issues and PRs from one repository can be added to multiple projects (e.g., a team backlog project and a release-specific project).

Recommended Management Practices

  • Creating and Managing Issues
  • Always open issues (tasks, bugs, features) directly in the relevant repository. This keeps discussions, code references, and history tied to the specific codebase.
  • Then, add those repository issues to one or more projects for higher-level planning and tracking. This effectively "links" the repository to the project(s) as needed.
  • Tracking Planned Feature Work vs Bug Fixes:
  • Use labels (e.g., feature, enhancement, bug, critical, good first issue, help wanted) in repositories to categorize issues.
  • In projects, create dedicated views, boards, or columns filtered by these labels—e.g., separate boards/sections for features (roadmap-focused) and bugs (triage/priority queue).
  • Prioritize bugs separately for quick fixes and stability; group features into epics, milestones, or iterations for planned releases.
  • Consider custom fields in projects (e.g., "Type: Feature/Bug", "Priority") for better sorting and automation.
  • Setting Up Labels and Milestones:
  • Labels (on the right sidebar) > New label or edit existing ones.
  • Recommended labels for volunteer-driven projects: bug (for defects), enhancement or feature (for new work), documentation (for docs updates), good first issue (easy tasks for newcomers), help wanted (open for contributions), priority:high/medium/low (for urgency), duplicate (to mark repeats), wontfix (for issues not to pursue).
  • Use colors for visual distinction (e.g., red for bugs, green for features).
  • Apply labels when creating issues, and use them in project filters for automated workflows.
  • Milestones: These group issues/PRs into releases or phases and are also repository-level. Set them up via Issues tab > Milestones (on the right) > New milestone.
  • Recommended for "when ready" releases: Create open-ended milestones like "Next Release" (no due date, for upcoming stable features), "Backlog" (for future ideas), or "v1.0" (for version-specific goals).
  • Assign issues to milestones when adding them to projects. Since no fixed sprints, use milestones flexibly to signal readiness rather than time-bound deadlines.
  • In projects, filter views by milestones for release planning.
  • Setting Up a New Project (Especially for Volunteer-Driven Work):
    1. In your organization or user profile, go to Projects and click New project.
    2. Choose a blank project or template (start with a Board for simple Kanban-style tracking).
    3. Name the project (e.g., "Roadmap and Backlog").
    4. After creation, go to project Settings (menu in top-right):Add a short description.
      • Add a short description.
      • Add a detailed README (supports Markdown) explaining:
        • Project overview and goals.
        • How volunteers can contribute (e.g., pick issues labeled good first issue or help wanted).
        • Contribution guidelines (link to CONTRIBUTING.md if available).
        • Note that this is volunteer-driven: no fixed sprints or deadlines; work progresses asynchronously and releases happen "when ready" once features are stable and reviewed.
        • Communication channels (e.g., issues, discussions).
        • Tips for newcomers: start small, ask questions in issues.
    5. Set up views:
      • Use a Board layout with columns like: New, To Do, In Progress, Review, Done.
      • In Board view, set up columns like New, To Do, In Progress, Review, Done.
      • Avoid time-bound iterations/sprints; instead, use status columns and priority sorting for asynchronous flow.
      • Add a Roadmap view for high-level feature planning without dates.
      • Make the project public or visible to encourage volunteer participation.
  • Creating a Project Template for Faster Setups:
    1. First, set up a complete project as described above, including views, custom fields, automations, and a README tailored for your volunteer workflow.
    2. Navigate to the project.
    3. Click the menu (three dots) next to the project name.
    4. Select Save as template (available for organization-owned projects).
    5. Provide a name and description for the template.
    6. To use it: When creating a new project, select New project > choose your template from the list. This copies the structure, views, fields, and automations for quick starts.
  • General Tips for Volunteer Projects:
    • Use organization-level projects for cross-repository oversight.
    • Label issues generously to attract contributors (e.g., good first issue for beginners).
    • Leverage automations, custom fields, and insights for efficiency.
    • Add clear READMEs to repositories and projects for context.
    • Emphasize patience and clear communication, as contributors work on their own time.
This approach ensures issues stay close to the code while projects provide flexible, high-level visibility—perfect for volunteer-driven open source without rigid schedules. For official details, see GitHub Docs on Projects and Repositories.
