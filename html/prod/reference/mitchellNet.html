MitchellNET — Home Network Design v2.0

This document is the master design specification for the MitchellNET home network rebuild. It covers architecture, component roles, VLAN segmentation, the test bench ethernet integration, bill of materials, deployment checklist, and rationale for all major decisions. Location: London, Ontario, Canada.

1. Architectural Overview

The design implements a high-performance, segmented network with two distinct planes:

  • 10Gbps Data Plane — connects the Mac Studio, NAS, and core infrastructure via SFP+ DAC cables through the MikroTik core switch for maximum throughput on video editing and storage workloads.
  • 1Gbps Management Plane (Out-of-Band) — a physically separate TP-Link switch provides a recovery lifeline to every device's management interface. Even if the core 10G network goes down or suffers a broadcast storm, the OOB rail remains reachable.
  • 1Gbps Lab Plane — a dedicated bench switch connects all four test instruments to the network, with a SCPI/LXI control path back to the Mac Studio on its own VLAN 50 segment.

Core services (DHCP, DNS, IDS/IPS, VLAN routing) live on the hardware firewall. Application-level services (MQTT, reverse proxy, monitoring) run as Docker containers on the Ubuntu iMac server.

3. VLAN Segmentation & WiFi SSID Mapping

Every device lands on exactly one VLAN. OPNsense is the gateway for all VLANs and enforces strict inter-VLAN firewall rules. The UniFi APs broadcast all SSIDs simultaneously — the SSID a device connects to determines its VLAN, regardless of which physical AP it uses.

  • VLAN 10 Management — 10.10.10.0/24. Infrastructure devices, OOB switch, management interfaces. Not reachable from other VLANs.
  • VLAN 20 Storage — 10.10.20.0/24. Mac Studio and NAS only. MTU 9000 (jumbo frames) for maximum NFS/SMB throughput.
  • VLAN 30 IoT — 10.10.30.0/24. ESP32 devices, smart home, seasonal devices (Christmas tree, candy cane pillars). Firewall allows only outbound to Mosquitto port 1883. Client isolation on the IoT SSID prevents device-to-device communication.
  • VLAN 40 DMZ — 10.10.40.0/24. Raspberry Pi server. Inbound HTTPS only from internet. Cannot initiate connections to LAN.
  • VLAN 50 Lab / Test Bench — 10.10.50.0/24. All four bench instruments (oscilloscope, signal generator, multimeter, power supply). Mac Studio permitted inbound SCPI connections on port 5025. No other LAN access.
  • VLAN 99 Guest — 10.10.99.0/24. Internet access only. Fully blocked from all internal resources. Maps to AgingApprentice-Guest SSID.

4. Test Bench — Ethernet Remote Control

All four bench instruments support remote control over ethernet using SCPI (Standard Commands for Programmable Instruments) over the LXI (LAN eXtensions for Instrumentation) standard. A dedicated unmanaged gigabit switch on the bench connects all instruments and uplinks to the MikroTik core switch on VLAN 50. The Mac Studio is permitted to reach instrument SCPI ports (TCP 5025) from its VLAN 20 address via a specific OPNsense firewall rule.

Software: LXI Tools

Install on Mac via Homebrew. Provides both a GUI and command-line tool for instrument discovery, SCPI commands, screenshots, and data logging. All four instruments are on the confirmed compatibility list. 

brew install lxi-tools

What Each Instrument Supports Remotely

  • Siglent SDS1202X-E (Oscilloscope) — capture screenshots to Mac, pull waveform data as files, set timebase/voltage/trigger remotely, automate measurements and log results, run headless data capture.
  • Siglent SDG-2042X (Signal Generator) — set frequency, amplitude, waveform type, and DC offset; configure sweep sequences; save and recall waveform setups.
  • Siglent SDM3055 (Multimeter) — READ? command returns live measurements; log voltage/current/resistance to CSV over time; automate repetitive measurements.
  • Rigol DP832 (DC Power Supply) — set voltage and current limits precisely; enable/disable outputs under software control; log actual output over time; automate power sequencing.

Python Automation

The pyvisa library allows Python scripts on the Mac to control all four instruments simultaneously for automated test sequences: 

pip install pyvisa pyvisa-py

Example use case — RC time constant experiment: a single script tells the SDG to output a 0.2Hz square wave, the DP832 to hold exactly 5.000V, the SDM to log capacitor voltage every 100ms, and the SDS to capture a screenshot at the end — all without touching any front panel.

5. Application Stack (Ubuntu iMac Server)

All services run as Docker containers on the 2019 iMac running Ubuntu Server.

  • Nginx Reverse Proxy — entry point for external web services; traffic isolated to DMZ VLAN 40.
  • Mosquitto MQTT Broker — logic hub for all IoT devices; firewall rules allow VLAN 30 devices to reach port 1883 only on this host.
  • Grafana + Prometheus + InfluxDB — monitoring dashboards; Prometheus scrapes device metrics; InfluxDB stores time-series data from instruments and IoT devices.
  • LibreNMS — polls the MikroTik switch and all network devices via SNMP over the OOB management rail; provides per-port traffic, SFP temperatures, and device health.

6. Firewall & Security

OPNsense is recommended over pfSense for this build. It has native Suricata integration with policy-based rule management, more frequent updates, a cleaner GUI, and stronger community development. Key security features:

  • Suricata IDS/IPS — runs in IPS mode with ET Open/Pro rules. Noisy rule categories (coin mining, DNS/TLS) are disabled to maintain throughput. Scheduled daily rule updates.
  • Stateful packet inspection — all inter-VLAN traffic goes through the firewall; default-deny with explicit allow rules per VLAN.
  • PPPoE passthrough — Bell HH3000 in bridge mode. OPNsense holds the public IP directly, eliminating double-NAT.
  • VPN — WireGuard for remote access; AES-NI hardware offloading on the Netgate 6100 keeps VPN overhead minimal.

Key Firewall Rules Summary

  • VLAN 20 (Storage) → Any: allow. Mac Studio and NAS communicate freely.
  • VLAN 30 (IoT) → Mosquitto port 1883 only: restricted. All other IoT traffic blocked.
  • VLAN 40 (DMZ) → LAN: blocked. Inbound HTTPS 443 from internet permitted; Pi cannot reach LAN.
  • VLAN 50 (Lab) → Mac Studio SCPI port 5025 only: restricted. Instruments isolated from all other network segments.
  • VLAN 99 (Guest) → LAN: blocked. Internet only.

9. Key Design Decisions — Rationale

MikroTik CRS317 vs. Enterprise Alternatives

Enterprise 16-port 10GbE switches (Cisco, Juniper, Arista) cost $3,000–$5,000 CAD and are loud enough to be a dealbreaker in a home office. The CRS317 delivers wire-speed 10GbE switching for ~$650 CAD with intelligent fan control (quiet or off under normal video editing loads), dual-boot SwOS/RouterOS flexibility, and deep SNMP telemetry ideal for Grafana dashboards. It draws 20–40W vs. 150W+ for older enterprise hardware — meaningful in Ontario where hydro costs are variable. Note: MikroTik's interface is engineer-first with a steeper learning curve than Ubiquiti. Familiarity with Docker and OPNsense helps significantly.

OPNsense vs. pfSense

OPNsense is the stronger choice for this build in 2025–2026: native Suricata integration with policy-based rule management, more frequent security updates, a modern GUI, and active community development without the commercial pressures that have complicated pfSense/Netgate's relationship with the community. Both are capable — pfSense remains solid if its documentation or WireGuard implementation is preferred.

DAC Cables vs. 10G-BaseT

Direct Attach Copper (DAC) cables have lower latency, consume less power, and stay cool compared to 10G-BaseT (RJ45) transceivers. RJ45 10GbE modules can reach temperatures that cause thermal throttling during long renders. For all rack-internal connections DAC is preferred; 10G-BaseT adapters are used only where a device (Mac Studio) lacks an SFP+ port.

Dedicated Bench Switch for Test Instruments

A separate switch for the test bench keeps instrument traffic isolated on VLAN 50 and provides a clean physical boundary between the lab and the main network. The OPNsense firewall enforces that only the Mac Studio can reach instrument SCPI ports, preventing any other device from accidentally interacting with the bench equipment. This also makes it easy to add future instruments without touching the core switch configuration.

10. Day 1 Deployment Checklist

Phase 1 — Bench Prep & Physical Labelling

  • Unbox and inspect all hardware. Verify power cables, mounting ears, and SFP+ modules.
  • Label every cable with a label maker before installation (e.g. CORE-TO-NAS, OOB-TO-FIREWALL, BENCH-TO-CORE).
  • Update firmware on each device (Firewall, Core Switch, OOB Switch) via existing internet before complex configuration begins.

Phase 2 — OOB Management Rail (Safety Net First)

  • Initialize TP-Link OOB switch. Set static management IP: 10.10.10.2.
  • Connect MikroTik console/MGMT port → OOB Switch.
  • Connect NAS IPMI port → OOB Switch.
  • Connect Firewall MGMT NIC → OOB Switch.
  • Connect Ubuntu iMac USB–Ethernet adapter → OOB Switch.
  • Test: laptop on OOB switch can ping all management interfaces. Do not proceed until all devices have a lifeline.

Phase 3 — Core VLAN & Data Plane

  • Define VLANs on OPNsense: VLAN 10 (10.10.10.1/24), VLAN 20 (10.10.20.1/24, MTU 9000), VLAN 30 (10.10.30.1/24), VLAN 40 (10.10.40.1/24), VLAN 50 (10.10.50.1/24), VLAN 99 (10.10.99.1/24).
  • Configure SFP+ firewall ↔ MikroTik port as VLAN trunk (tagged for all VLANs).
  • Set MikroTik ports for NAS and Mac Studio to access mode — untagged on VLAN 20.
  • Set MikroTik port for bench switch to access mode — untagged on VLAN 50.
  • Configure MikroTik trunk port to UniFi APs (tagged for VLANs 10, 30, 99).

Phase 4 — ISP Integration

  • Input Bell b1xxxxxx PPPoE credentials into OPNsense WAN interface.
  • Verify OPNsense shows a public IP address (not 192.168.x.x — that means double-NAT).
  • Disable Bell HH3000 WiFi, DHCP, and UPnP completely.

Phase 5 — Service Migration

  • Assign Ubuntu iMac a static IP on VLAN 10 (management).
  • Verify Nginx serves one test site reachable from the internet via public IP.
  • Verify LibreNMS successfully polls MikroTik via SNMP over OOB rail.
  • Verify Grafana/Prometheus dashboards are reachable from Mac Studio.

Phase 6 — Test Bench Integration

  • Enable DHCP on each instrument: Utility → I/O → LAN → DHCP on.
  • Note each instrument's assigned IP from its front panel menu.
  • Verify each instrument's built-in web interface loads in Mac browser.
  • Install LXI Tools on Mac: brew install lxi-tools.
  • Run LXI Tools discovery scan — all four instruments should appear.
  • Add OPNsense firewall rule: VLAN 20 (Mac Studio) → VLAN 50 (instruments) port 5025 allow.

11. Risks & Key Considerations

  • Bell HH3000 bridge mode — must be in true PPPoE passthrough; otherwise double-NAT will break VLAN firewall setup and external connectivity.
  • NAS throughput — ensure SSD cache tier or RAID with fast drives for sustained 10Gbps editing. Without caching, spinning disks will bottleneck well below 10G.
  • Suricata throughput — IDS/IPS in full inspection mode can reduce throughput on weaker hardware. Tune noisy rule categories (coin mining, DNS, TLS) to maintain line rate on the Netgate 6100.
  • MikroTik learning curve — RouterOS is engineer-first. Start with SwOS for simplicity; migrate to RouterOS when comfortable with VLAN and routing concepts.
  • IoT device MAC tracking — seasonal devices (Christmas lights, candy cane pillars) will get new DHCP leases each season. Track devices in Grafana/LibreNMS by MAC address, not IP.
  • Log storage for SNMP + IDS — plan log rotation. SNMP polling + Suricata alert logs can consume storage quickly without retention policies in InfluxDB/Prometheus.
  • Ontario hydro costs — 24/7 rack power draw matters. CRS317 (~30W) + Netgate 6100 (~15W) + TP-Link (~5W) + Pi (~7W) ≈ 57W continuous. NAS and iMac server dominate total draw.
